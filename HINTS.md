## Часть 1. Классы и модификаторы доступа

### 1.1 `BankAccount`

- **Что повторить:**  
  - модификаторы доступа (`private`, `public`, `static`);  
  - статический блок инициализации;  
  - блок инициализации экземпляра;  
  - `toString()` и форматированный вывод (`String.format`).
- **На что обратить внимание в коде:**
  - как используется `totalAccounts` для подсчёта созданных счетов;
  - как формируется `accountNumber` на основе счётчика;
  - почему валидация на `amount <= 0` и недостаток средств делает код более «безопасным».
- **Типичные ошибки:**
  - прямой доступ к полям (`a1.balance = -100`) вместо использования методов;
  - отсутствие проверки на отрицательные суммы;
  - неиспользование `static` для общих полей (например, `bankName`).

### 1.2 `Employee` / `HRSystem`

- **Что повторить:**  
  - различия между `public`, `protected`, package-private (без модификатора) и `private`;  
  - видимость полей/методов в разных пакетах.
- **Подсказка по таблице A–H:**  
  - спросите себя для каждой строки: «откуда идёт доступ? тот же ли это пакет? является ли класс наследником?».
- **Почему появился `EmployeeFixed`:**
  - пример правильной инкапсуляции: поля `private`, публичные геттеры и методы;
  - метод `authenticate` скрывает детали проверки пароля — безопаснее, чем прямой доступ.

### 1.3 `VarDemo`

- **Что повторить:**  
  - правила использования `var` в Java: только локальные переменные, нужна инициализация, нельзя поля/параметры.

## Часть 2. Абстрактные классы, sealed, интерфейсы

### 2.1 Иерархия `Employee` (бонусы)

- **Что понять из кода:**  
  - почему базовый класс `Employee` — абстрактный;  
  - как `totalCompensation()` использует полиморфизм: вызывает `calculateBonus()` нужного подкласса;  
  - как форматируется строка в `toString()`.
- **Упражнение:**  
  - добавьте новый тип сотрудника (например, `SeniorDeveloper`), который считает бонус по другой формуле, но **не меняйте** код в `EmployeeBonus.main` — он должен заработать автоматически.

### 2.2 `PaymentMethod` (sealed interface)

- **Ключевые идеи:**  
  - `sealed interface` ограничивает список реализаций (`permits`);  
  - записи (`record`) уже дают `equals`, `hashCode`, геттеры;  
  - `PaymentProcessor.describe` показывает, как различать реализации через `instanceof`.
- **Типичная ошибка:**  
  - пытаться добавить новый тип платежа, **не перечислив** его в `permits` — это не скомпилируется.

### 2.3 `Loggable`

- **Что потренировать:**  
  - `default` методы — общий код для всех реализаций;  
  - `static` метод интерфейса (`getLogLevel()`);  
  - `private` метод для повторно используемой логики (форматирование времени).
- **Безопасность:**  
  - логируйте только то, что можно показывать пользователю; пароли и секреты логировать нельзя (запомните это правило на будущее).

## Часть 6. Аннотации и Reflection

### 6.1 `ValidationFramework`

- **Что понять:**  
  - как аннотации `@NotEmpty` и `@Range` описывают требования к полям;  
  - как `Validator` через Reflection читает аннотации и значения полей.
- **Безопасность:**  
  - Reflection даёт много возможностей, но легко нарушить инкапсуляцию. В реальных проектах старайтесь использовать проверенные библиотеки (`Bean Validation`, `Hibernate Validator` и т.п.).

## Часть 7. Анонимные классы, лямбды, ссылки на методы

### 7.1 RefactorOriginal / RefactorStep1 / RefactorStep2

- **План работы:**  
  1. В `RefactorOriginal` найдите все анонимные классы.  
  2. В `RefactorStep1` замените их на лямбды с тем же типом (`Comparator`, `Consumer`, `Function`, `Predicate`, `Supplier`).  
  3. В `RefactorStep2` там, где лямбда просто вызывает существующий метод, используйте ссылку на метод.
- **Вопросы для самопроверки:**  
  - почему `city -> System.out.println(city)` можно заменить на `System.out::println`;  
  - почему `s -> s + \"!\"` **нельзя** заменить на ссылку на метод.
