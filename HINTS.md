## Часть 1. Классы и модификаторы доступа

### 1.1 `BankAccount`

- **Что повторить:**  
  - модификаторы доступа (`private`, `public`, `static`);  
  - статический блок инициализации;  
  - блок инициализации экземпляра;  
  - `toString()` и форматированный вывод (`String.format`).
- **На что обратить внимание в коде:**
  - как используется `totalAccounts` для подсчёта созданных счетов;
  - как формируется `accountNumber` на основе счётчика;
  - почему валидация на `amount <= 0` и недостаток средств делает код более «безопасным».
- **Типичные ошибки:**
  - прямой доступ к полям (`a1.balance = -100`) вместо геттеров/сеттеров или предметных методов (нарушение инкапсуляции и сокрытия информации);
  - отсутствие проверки на отрицательные суммы или недостаток средств;
  - неиспользование `static` для полей, общих для всех экземпляров (например, `bankName`).

### 1.2 `Employee` / `HRSystem`

- **Что повторить:**  
  - различия модификаторов доступа: `public`, `protected`, доступ в пределах пакета (package-private, без модификатора), `private`;  
  - видимость полей и методов из другого пакета и в иерархии наследования.
- **Подсказка по таблице A–H:**  
  - спросите себя для каждой строки: «откуда идёт доступ? тот же ли это пакет? является ли класс наследником?».
- **Почему появился `EmployeeFixed`:**
  - пример корректной инкапсуляции: поля `private`, доступ только через публичные геттеры и методы;
  - метод `authenticate` инкапсулирует проверку пароля (сокрытие информации) — безопаснее, чем прямой доступ к полям.

### 1.3 `VarDemo`

- **Что повторить:**  
  - правила вывода типа `var` в Java: только для локальных переменных, обязательна инициализация при объявлении; нельзя использовать для полей класса и параметров методов.
- **Совет:**  
  - попробуйте добавить свои примеры (например, с `Map<String, List<Integer>>`) и посмотреть, как компилятор выводит тип.

## Часть 2. Абстрактные классы, sealed, интерфейсы

### 2.1 Иерархия `Employee` (бонусы)

- **Что понять из кода:**  
  - почему базовый класс `Employee` объявлен абстрактным (есть абстрактный метод, экземпляры не создаются);  
  - как `totalCompensation()` использует полиморфизм: вызов `calculateBonus()` связывается с реальным типом объекта (динамическое связывание);  
  - как форматируется строка в `toString()`.
- **Упражнение:**  
  - добавьте новый тип сотрудника (например, `SeniorDeveloper`), который считает бонус по другой формуле, но **не меняйте** код в `EmployeeBonus.main` — он должен заработать автоматически.

### 2.2 `PaymentMethod` (sealed interface)

- **Ключевые идеи:**  
  - sealed-интерфейс ограничивает круг допустимых реализаций (ключевое слово `permits`);  
  - тип `record` автоматически предоставляет канонический конструктор, компонентные аксессоры, `equals`, `hashCode`, `toString`;  
  - в `PaymentProcessor.describe` демонстрируется различение реализаций через `instanceof` (и при необходимости pattern matching).
- **Типичная ошибка:**  
  - пытаться добавить новый тип платежа, **не перечислив** его в `permits` — это не скомпилируется.

### 2.3 `Loggable`

- **Что потренировать:**  
  - методы с реализацией по умолчанию (`default`) — общий код для всех реализаций интерфейса;  
  - статический метод интерфейса (`getLogLevel()`);  
  - закрытый метод интерфейса (`private`) для внутренней вспомогательной логики (например, форматирование времени).
- **Безопасность:**  
  - в логи попадают только данные, допустимые для разглашения; пароли, токены и секреты логировать запрещено.

## Часть 6. Аннотации и Reflection

### 6.1 `ValidationFramework`

- **Что понять:**  
  - аннотации `@NotEmpty` и `@Range` задают декларативные ограничения на поля;  
  - класс `Validator` с помощью Reflection получает метаданные аннотаций и значения полей и проверяет их соответствие.
- **Безопасность и корректность:**  
  - Reflection даёт доступ к полям и методам с обходом модификаторов доступа, что ослабляет инкапсуляцию. В промышленной разработке обычно используют готовые библиотеки валидации (например, Bean Validation, Hibernate Validator).

## Часть 7. Анонимные классы, лямбды, ссылки на методы

### 7.1 RefactorOriginal / RefactorStep1 / RefactorStep2

- **План работы:**  
  1. В `RefactorOriginal` найдите все анонимные классы, реализующие интерфейсы с одним абстрактным методом.  
  2. В `RefactorStep1` замените их на эквивалентные лямбда-выражения (типы: `Comparator`, `Consumer`, `Function`, `Predicate`, `Supplier`).  
  3. В `RefactorStep2` там, где лямбда лишь передаёт аргумент в один существующий метод, замените её на ссылку на метод.
- **Вопросы для самопроверки:**  
  - почему выражение `city -> System.out.println(city)` можно заменить на `System.out::println` (сигнатура совпадает с `Consumer.accept`);  
  - почему `s -> s + "!"` нельзя выразить ссылкой на метод (нет готового метода «строка + суффикс» с одной переменной).
